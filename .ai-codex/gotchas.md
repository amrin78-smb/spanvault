# SpanVault gotchas — read before touching related code

Dense list of non-obvious behaviours. Full detail lives in `CLAUDE.md` (search
sections by name below) and inline code comments — this is a pointer list, not
a replacement.

## Architecture / auth
- No local login page. Auth is 100% via the NocVault hub SSO redirect -> `/sso` -> this app's OWN `POST /api/sso` (Express, NOT a Next.js route) -> hub `sso-verify` called server-to-server (a direct browser fetch is CORS-blocked; bit us in 1.71.3/1.71.4).
- `frontend/src/app/api/**` routes other than the NextAuth catch-all are DEAD CODE — `middleware.ts` rewrites every non-`/api/auth/*` request straight to Express BEFORE Next.js routing sees it. A `frontend/src/app/api/sso/route.ts` was added in 1.71.3 (copying DDIVault's pattern), built clean, and never ran at runtime; fixed in 1.71.4 by moving the logic into `api/server.js`. DDIVault's equivalent pattern (next.config.js rewrite allowlist) is NOT transferable here — 4 different `/api/*` architectures exist across the NocVault suite apps.
- Adding a new intentionally-public route needs THREE independent gates cleared, each in a different file, each only surfacing once the previous is fixed (took 3 broken releases in a row for `/api/sso`): (1) `middleware.ts`'s `PUBLIC_API` regex, (2) `api/server.js`'s `WRITE_GATE_EXEMPT` array (only matters for non-GET), (3) `api/server.js`'s `enforceLicense` — TWO separate checks (`isAck`/`isSso` write-block exemption AND the `exemptPaths` disabled-state array).
- `getServerUrl(req)` (api/server.js, used for remote-agent install/dial-back URLs) and `resolveOrigin(req,...)` (used for browser-facing links/hub proxying) have DELIBERATELY OPPOSITE priority: `getServerUrl` checks `SV_PUBLIC_URL` env var FIRST, falling back to the request's Host header; `resolveOrigin` derives from the request Host FIRST, env var only as last-resort. Do not "fix" one to match the other — an installed remote agent must keep dialing the address it was configured with even if an admin later browses via a different hostname; a browser link should always follow the current request's hostname.
- `middleware.ts`'s per-user app-access gate (`appAllowed`) must run on BOTH the page-render branch AND the `/api/*` proxy branch — a gate on only one leaves a denied user's still-valid session able to pull data straight from the API. Release 1.71.1's notes claimed this was done; the commit never touched `middleware.ts`; the real fix didn't land until 1.71.7. Lesson: when a release note claims an enforcement fix, `git show --stat` the commit and re-read the live file — don't trust the notes.
- `/api/internal/*` (currently just `POST /api/internal/agents/push-config`) is blocked outright by `middleware.ts` (never proxied) AND gated to loopback-only inside Express (`requireLoopback`) — belt-and-suspenders because Express alone can't distinguish "the frontend proxy called on behalf of a browser" from "the collector called directly" — both arrive from 127.0.0.1.

## RBAC / site-scoping
- Site-scoping was found MISSING on a batch of similar routes even though a sibling route added in the same commit had it right: `GET /api/service-checks/:id` and `/:id/results` shipped with zero `getSiteFilter` check while `/api/reports/service-detail` in the same diff had it correctly. All three are now fixed. Rule when reviewing a new batch of routes for one resource: check EACH one individually, don't spot-check one and assume the rest match.
- `GET /api/wireless/aps/:id` was missing `getSiteFilter` scoping until the 2026-07-22 bug sweep, even though the sibling list endpoint (`/api/wireless/aps`) and `/api/wireless/aps/:id/clients` both already had it. Made more reachable that day by an Alerts-page deep link giving every role (including site_admin) a prominent UI path straight into this route by AP id.
- Also fixed in the same class of bug (2026-07 security fix): `GET /api/wireless/clients/:mac` and `/:mac/history` had zero site-scoping, letting a site_admin read another site's wireless client bandwidth/RSSI/connection-event history by MAC or AP id.
- `getSiteFilter(req)` reads `x-user-sites`/`x-user-role` headers set by the frontend proxy from the verified session — a `site_admin` is scoped, admin/viewer/missing headers get `null` (no filter). Enforced server-side in Express, not just client-side, so a site_admin can't bypass it by calling the API directly.

## Wireless collector / RF specifics
- `testController(pool, controller)` is commented "dry run — no DB writes" and IS true for SNMP controllers, but is DELIBERATELY FALSE for API controllers with a rotating credential (currently just `aruba_central`): it forwards `pool` into `pollApiController` so a refreshed OAuth token gets persisted. Aruba Central's `refresh_token` ROTATES on every use — if the newly-issued token from a "Test Connection" click isn't saved, the only refresh_token left in the DB is invalidated, permanently bricking the integration (fixable only by a human re-authorizing in the Central UI). Don't drop the `pool` argument to make the "dry run" comment literally true.
- Aruba Central's per-AP RF data (channel/util/tx-power/noise-floor) comes from a SEPARATE, slower poll cycle (`pollRf`, default 900s/15min, `SV_ARUBA_RF_POLL_INTERVAL_SECONDS`) — NOT the main 5-min AP poll, which deliberately never reports RF for aruba_central (rate-limit budget: ~2300 calls/day on the main cycle vs ~770/day on its own, staying under Central's ~5000/day cap). Never fold this back into the main poll.
- Corollary: `upsertAp()`'s RF columns (`radio_2g/5g_channel`, `_util_pct`, `tx_power_2g/5g`, `noise_floor_2g/5g`) are written via `COALESCE($n, existing_column)`, never a plain overwrite — because the main poll always contributes `null` for aruba_central's RF fields. Without the COALESCE, the next main poll (5 min later) would silently wipe out what the 15-min `pollRf` cycle just wrote. Never "clean up" these into plain assignments.
- `pollRf` never refreshes the OAuth token itself — it reuses whatever the main `poll()` last persisted and skips the cycle if the token is missing/expired. A second independent refresher here would risk racing two refreshes inside Central's ~15-min refresh window.
- Wireless RF alerting (`evaluateWirelessAlerts()` in collector.js) is WINDOWED/SUSTAINED, not single-poll: `wireless_high_util` compares a rolling-window average, not one poll's reading, because a genuinely busy AP's utilization swings poll-to-poll (9-73% observed) and rarely lands above a single-poll cliff. 7 checks total, each independently configured via `settingInt()`/`setting()` (14 `app_settings` keys, code-default-only, NO Settings-page UI): `wireless_high_util` (warn/crit two-tier), `wireless_high_retry`, `wireless_client_imbalance`, `wireless_high_interference`, `wireless_degraded_noise_floor`, `wireless_roam_storm`, `wireless_weak_clients`.
- `wireless_client_imbalance` only evaluates a band the AP actually has a radio for (channel not null) — never flag a 0-client band the AP doesn't physically have.
- `wireless_degraded_noise_floor`'s comparison is inverted and easy to get backwards: noise floor is negative dBm, "degraded" means the rolling average is `>=` the threshold (LESS negative = worse; -80dBm is worse than -95dBm).
- Escalating a two-tier alert (e.g. `wireless_high_util` warn->crit) needs resolve-then-re-raise, NOT a second `raiseWirelessAlert` call — the `alerts` table's partial unique index (one ACTIVE row per ap+type) means `ON CONFLICT DO NOTHING` silently no-ops a second raise at a higher severity while the warn-tier row is still active, permanently capping the alert at whichever severity fired first.
- `computeCongestionScore()` (collector/wirelessScore.js, pure function, no DB access) blends util(35%)/retry(25%)/interference(15%)/imbalance(15%)/weak-client-ratio(10%) into a 0-100 score + low/medium/high level. Fixed in the 2026-07-22 bug sweep: the ORIGINAL 70/40 level cutoffs meant util saturated at 100% (worth only 35 of its own weight) could never alone escape "low" — directly contradicting the file's own "util dominates" comment. Cutoffs are now 60/35, derived by hand against what the weighted components can actually produce (util alone maxed=35 -> at least medium; util+retry both maxed=60 -> high). Re-derive by hand again if the weights ever change.
- Congestion score is DISPLAY-ONLY, computed live at read time in `GET /api/wireless/aps` and `/aps/:id` — nothing in the collector persists it, no new column on `wireless_aps`. The API's rolling window is a FIXED 15-minute `INTERVAL`, not a read of the `wireless_util_window_minutes` setting (deliberate simplification) — if that setting is ever changed from its default, the score's window silently drifts out of sync with the alerts' window. `congestion_score`/`congestion_level` are always `null` together for a non-`online` AP, never a misleading `0`/`'low'`.
- `wireless_clients` rows are HARD-DELETED after 15 min of inactivity and get a brand-new `id` on reconnect — never FK a history/alert table to `wireless_clients.id`; key off `(mac_address, controller_id)` instead (this is why `wireless_client_history` and the `wireless_client_bandwidth_high` alert both use mac+controller, not a client row id).
- Client-poll counters (`prevClientCounters`) live in a SEPARATE Map from AP-poll counters, pruned on the CLIENT_POLL_INTERVAL (10min), not the AP one (5min) — reusing the AP map would evict a client's counter baseline moments before its own next poll needed it.
- `wrapAwareDelta`/`counterDeltaBps` apply a sanity cap (`MAX_SANE_BPS = 2e9`) on EVERY delta branch, not just the Counter32-wrap-recovery branch — a collector-restart timing artifact on the "normal" branch alone previously produced one-off multi-hundred-Gbps readings before self-correcting on the next poll.
- Client sustained-bandwidth alerting requires TWO consecutive over-threshold polls (in-memory debounce, `clientBandwidthBreaches` Map) before raising — one large file transfer on a single poll must not alert; a collector restart just resets the counter (delays, never falsely raises).
- Rogue/unmanaged AP detection, HA peer roster (`ha_peers`), and chassis-temp columns are all live-verified against real Aruba hardware per inline comments — don't assume a field name is speculative without checking the comment first.

## Reports / scheduling
- `GET /api/reports/pdf/:template` only accepts templates with a real pdfkit renderer in `api/reportsPdf.js` (`hasPdfRenderer`) — everything else 404s; DB/stack errors are never leaked to the client.
- Two independent PDF/report code paths exist by design: `api/reportsPdf.js` (pdfkit, server-side, used for on-demand PDF download AND scheduled emails via `api/reportScheduler.js`) vs the `frontend/src/components/reports/*Report.tsx` renderers (browser `window.print()` path for the interactive Reports page). This is a deliberate two-path split, not duplication to consolidate (see the analogous documented decision for SpanVault's own reporting hybrid).
- `api/reportScheduler.js`'s `runAndEmailReport` never uses nodemailer's `raw` option — load-bearing for the deferred nodemailer v6->v9 security-debt decision in CLAUDE.md (the v6 CVE requires the `raw` option to be reachable).

## Testing
- `tests/` is OFFLINE PARSER UNIT TESTS ONLY — no framework (plain Node scripts, `spawnSync` per suite), no network, no DB. Run via `node tests/run-parser-tests.js` or `npm run test:parsers`; exits non-zero on any failure (CI-gateable). Covers `collector/wireless/*` vendor SNMP parsers (Aruba, Cisco, Ruckus, Mikrotik, HPE, Fortinet, Grandstream) against synthetic MIB-shaped walk data. Run this after ANY change to `collector/wireless/*` or `collector/snmp-session.js`.
- Playwright is NOT a project dependency (not in package.json, no config file anywhere in the repo) despite commit messages referencing "confirmed live via a Playwright screenshot" — those refer to AD HOC manual UI verification during development sessions (e.g. an MCP browser tool), not a checked-in test suite. Don't go looking for a `playwright.config.ts` or `e2e/` directory; there isn't one.

## Schema / privilege
- `scripts/schema.sql` is idempotent and RE-APPLIED ON EVERY API STARTUP (since 1.25.2, via advisory lock) — not just at install time. See `.ai-codex/schema.md`'s "Known schema debt" for a specific, concrete concern: this file contains an unconditional `GRANT SELECT ON ALL TABLES IN SCHEMA public TO nocvault_readonly`, which appears to contradict CLAUDE.md's explicit warning that this exact blanket form must never be re-run (it would re-widen `wireless_controllers`' deliberately-excluded secret columns). Verify where the column-level exclusion actually lives before touching either this GRANT block or the readonly-role grant story.
- Never re-add a `host()`/`inet` cast on `netvault.devices.ip_address` when joining/importing — it's `character varying`, not `inet` (confirmed 2026-07 against live `information_schema.columns`; a cast here previously broke NetVault sync/import in three places).
- `wireless_controllers` holds 5 live secret columns (`api_key`, `api_password`, `api_client_secret`, `api_refresh_token`, `api_access_token`) — must always be column-level-excluded from any readonly diagnostic role, never table-wide granted.

## Frontend conventions
- No sub-components defined inside another component's function body anywhere in the codebase currently (verified — see `.ai-codex/components.md`'s Violations section) — keep it that way; it causes input focus loss on every keystroke via remount-on-render.
- `api/` and `collector/` are plain JavaScript, zero TypeScript syntax (no `as string`, no type annotations) — only `frontend/src/` uses TypeScript.
- `frontend/next.config.js` has no `output: 'standalone'` — the NSSM service runs `next start -p 3008` directly; don't add standalone-build assumptions (static copying etc.) without checking this first.
- Design tokens (`--text-*`, `--bg-*`, `--tint-*`, `--surface-subtle`) are the suite-wide standard defined in `frontend/src/app/globals.css` — SpanVault is the REFERENCE implementation other NocVault apps copy from, not a follower. Hardcoded hex duplicating a token breaks dark mode. Sticky headers/toolbars must use an OPAQUE `var(--bg-card)`/`var(--bg-primary)`, never a semi-transparent tint — scrolled content bleeds through otherwise (worst in dark mode).
